## Project Background & Purpose

### Overview / Elevator Pitch

Many students want to study better, but not everyone has a simple way to track time and progress. StudyWave was built as a simple browser tool for logging study sessions and checking progress.

It was also a practical way for me to improve frontend, UI design, and project planning skills.

### Goals & Objectives

- Build a useful tool for students by delivering all core StudyWave features by April 30, 2026.
- Complete the full project lifecycle (planning, design, development, testing, documentation, submission) in one end-to-end project.
- Improve frontend skills by building the app only with HTML, CSS, and JavaScript (no frameworks).
- Keep the app simple and easy to use by ensuring all main actions can be completed in 3 clicks or less.
- Launch a responsive version that works cleanly on mobile (768px and below) and desktop before final submission.

## Risk & Issue Management

### Initial Risks

1. Project scope becoming too big.
2. Managing time with other coursework.
3. Bugs during final testing.
4. UI problems on small screens.
5. Limited time for extra features.

### Management Approach

Risks and issues are logged in a simple tracker (Notion) with a short description, impact level (low/medium/high), and action owner (me). I review risks weekly, raise urgent issues immediately, and resolve high-impact items before working on optional features.

### Assumptions

- Users have access to a web browser.
- The project can be managed by one person.
- Required tools stay available.
- Main requirements do not change too much.
